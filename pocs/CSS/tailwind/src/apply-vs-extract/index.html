@apply, @layer, and when to extract

Same button. Three models. Notice what breaks when a designer asks for a one-off variant.

1 · @apply (outside @layer)

Override loses. Unlayered styles beat all layered utility classes — bg-red-500 never fires.

@reference "tailwindcss"; /* v4 */
.btn-primary {
  @apply px-5 py-2.5 rounded-lg
    bg-indigo-500 text-white
    hover:bg-indigo-600 ...;
}

2 · @apply in @layer components

Override wins. @layer utilities outranks @layer components. But now the HTML is doing CSS work anyway.

@layer components {
  .btn-layer {
    @apply px-5 py-2.5 ...;
  }
}

3 · Inline utilities (component)

No override needed. Variants are props. The component owns the mapping — zero CSS written per variant.

// React
<Button />
<Button variant="danger" />
// resolves to a different
// class string, not new CSS

The one-off variant problem

Designer wants a teal CTA on one page. What does each model cost?

@apply: write a new rule

Required: .btn-teal { @apply bg-teal-500; } — CSS grows with every designer request.

@layer: override inline

Needed: class="btn-layer bg-teal-500 hover:bg-teal-600" — back to writing utilities. So why extract at all?

Component: a prop

Needed: <Button color="teal" /> — zero CSS. Propagates everywhere. One place to change.

Verdict: @apply outside @layer creates specificity landmines. Inside @layer components is better but every override pushes you back to writing utilities. The component model is the answer — variants become props, not CSS rules, and refactors touch one file.
