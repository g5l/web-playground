Variant Prefixes

Stack hover:, focus-visible:, md:, dark:, disabled:, aria-*, data-[*]:, and [&>p]: on one element.

The ordering rule

Responsive prefixes produce @media wrappers. State prefixes produce pseudo-selectors. Their position in the class string is convention only — Tailwind always emits correct CSS regardless of order.

State only

hover:bg-indigo-500

→ CSS

.hover\:bg-indigo-500:hover { background-color: ... }

Responsive wraps state

md:hover:bg-violet-500

→ CSS

@media (min-width: 768px) {
  .md\:hover\:bg-violet-500:hover { background-color: ... }
}

dark: wraps state

dark:hover:bg-slate-700

→ CSS

:where(.dark, .dark *) .dark\:hover\:bg-slate-700:hover { ... }

Arbitrary data attribute

data-[state=open]:opacity-100

→ CSS

.data-\[state\=open\]\:opacity-100[data-state="open"] { opacity: 1 }

Convention: base → state → responsive. Write hover:bg-indigo-500 md:hover:bg-violet-500, not the reverse. The class string reads "what happens, then where it applies."

aria-expanded:

JS sets the attribute; Tailwind's variant fires. No class list manipulation. The label color uses aria-expanded: directly on the button. The chevron uses group-aria-expanded: inherited from the parent.

Content revealed. The button's aria-expanded attribute drove both its own label color and the chevron rotation — zero class-list changes in JS.

data-[state=open]:

Arbitrary data-attribute variants let you mirror any state from your JS or headless UI library directly into CSS — no conditional class strings needed.

Panel is open

Only data-state changed in JS. Opacity, transform, and pointer-events all followed from the Tailwind variants. 

opacity-0 translate-y-2 pointer-events-none transition-all

data-[state=open]:opacity-100
data-[state=open]:translate-y-0
